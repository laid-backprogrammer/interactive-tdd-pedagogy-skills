# Interactive TDD Pedagogy 中文说明

`interactive-tdd-pedagogy` 是一个用于 AI 结对编程的 agent skill。它的目标不是让 AI 替你写更多代码，而是让你在 AI 写代码越来越快的情况下，仍然保持对项目行为、测试证据和长期维护责任的掌控。

## 它解决什么问题

现在模型已经可以一次生成成百上千行代码，产品需求也在持续变化。我们不可能完全回到过去那种“古法编程”，每一行都慢慢手写、慢慢摸索。

但另一个风险也很明显：AI 写得越多，开发者越容易只看结果，不再真正理解输入、输出、状态变化、失败路径和业务规则。久而久之，代码库会变成黑盒。项目看起来推进得更快，但维护、调试、评审和承担责任会变得更难。

这个 skill 的核心目标就是反过来使用 AI：让 AI 成为代码阅读、行为理解和项目接手的教练，而不是让 AI 把项目变成更大的黑盒。

## 核心观点

TDD 的精髓不是“为了测试而测试”，也不是形式上必须先写测试。TDD 真正有价值的地方在于：测试迫使我们理解代码行为。

也就是：

- 输入是什么；
- 输出应该是什么；
- 状态应该如何变化；
- 失败路径是什么；
- 哪些业务规则必须被保留；
- 哪些行为已经被证明，哪些还没有被证明。

只有真的理解输入输出和代码行为，才可能写好代码、评审代码、维护代码，也才可能在 AI 生成代码后判断它到底是不是可靠。

## 它怎么工作

这个 skill 会把项目接手拆成很小的循环：

1. 选一条很窄的业务路径，而不是解释整个项目。
2. 建立一个 Mini Behavior Contract，先明确这一轮要理解的行为。
3. 从一个测试断言开始，反向追到业务代码。
4. 让你预测一次输入、输出或状态变化。
5. 如果要进入 TDD，先确认 Red，再做最小 Green。
6. 记录已经证明的行为、还没证明的行为、后续要复习的点。

它会刻意避免一次性讲完整模块、完整框架或完整语言语法。每一轮只推进一个很小的判断点，让你能跟上，也能逐步形成自己的代码阅读能力。

## Mini Behavior Contract

每一轮都会先建立一个小契约：

```text
当 <输入或触发条件>
系统应该 <可观察输出>
并且应该改变 <状态>
因为 <业务规则>
```

没有行为契约，就不进入 Red；没有 Red，就不进入修复；行为没有保持不变，就不能叫重构。

## Harness 的意义

Harness 一方面让 agent 和期望对齐，另一方面也让开发者自己和期望对齐。

这很重要。因为 AI 可以帮我们写代码，但真正对项目负责的人仍然是开发者。LLM 再强，本质上仍然是概率模型。它能提高速度，但不能替代我们对项目行为的责任。

如果 harness 只被理解成“让 agent 通过测试”，那是不够的。更好的理解是：harness 是一套可执行的共同理解。它让人、测试和 agent 围绕同一套行为期望工作。

长期来看，这能防止 AI 辅助开发变成“越写越黑盒”，也能防止开发者的代码阅读能力退化成完全依赖 AI。

## 适合什么时候用

适合这些场景：

- 你要快速接手一个陌生项目；
- 你不熟悉当前项目使用的语言或框架；
- 你想通过测试理解真实业务行为；
- 你想评审 AI 写出来的代码，而不是盲信；
- 你想把一个模糊需求变成可观察、可验证的行为；
- 你想在 AI 结对编程中提升自己的代码阅读能力和项目掌控力。

## 一句话总结

这个 skill 的目标是：让 AI 结对编程不只是提高写代码速度，也提高开发者对代码行为的理解、对新项目的接手能力，以及对长期工程责任的掌控力。
